# 运行前提与最小配置

## 目的

这个文档说明：要让 `stock-agent-orchestrator` 真正作为一个多 agent 协同选股框架运行，前提需要准备什么。

核心结论：

- 只跑本地预检，不需要真实 agent 和飞书群。
- 要验证产品价值，必须接入飞书。
- 要跑完整透明协同，至少需要 `1 个用户 + 3 个 agent + 1 个飞书群 + 1 个后台 orchestrator`。

## 最小角色配置

### 1. 用户

职责：

- 在飞书群里委托任务
- 审批新规则和越界动作
- 核实最终结果是否闭环

必须存在。

### 2. 小C

定位：任务 owner。

职责：

- 接收用户委托
- 建任务
- 分派给小智 / 小巴
- 追办状态
- 检查证据是否落盘
- 判断是否进入 `WAITING_USER`
- 收口并同步结果

完整运行时必须存在。

### 3. 小智

定位：数据层 agent。

职责：

- 拉取七层数据
- 补充行情、RSI、财务、资金面、催化剂、题材、风险点
- 提供候选池所需的数据证据
- 回填数据状态

完整运行时必须存在。

早期可以用本地样本或手动文件临时代替，但这只能算预检。

### 4. 小巴

定位：分析层 agent。

职责：

- 根据数据做筛选和分析
- 输出候选池判断
- 输出单票建仓 / 观望 / 放弃建议
- 解释规则依据
- 参与复盘和规则沉淀

完整运行时必须存在。

早期可以用本地样本或手动文件临时代替，但这只能算预检。

### 5. Orchestrator

定位：后台任务状态机。

职责：

- 把飞书消息转成任务
- 维护任务状态
- 记录每一步事件
- 判断当前责任人
- 检测静默断链
- 输出任务卡和状态回执
- 连接候选池、七层研究卡、规则记忆

这是本仓库提供的主体能力。

## 最小平台配置

### 飞书群

必须有一个群作为透明控制面。

推荐分两类：

- `beta 测试群`：先用于主动测试，不影响正式流程。
- `正式群`：稳定后灰度接入。

不建议一开始直接让新框架在正式群主动发言。

### 飞书机器人或消息通道

至少需要满足两种能力：

- 读取群消息：用于识别委托和状态回执。
- 发送群消息：用于 beta 阶段展示任务卡、追办、审批请求和结果收口。

阶段差异：

- Shadow 阶段：只需要读取，不发送。
- Beta 阶段：需要读取和发送。
- 正式灰度：需要严格控制发送策略和审批边界。

### 任务数据库

需要一个持久化位置保存任务状态。

当前 MVP 使用 SQLite。

后续可扩展到：

- PostgreSQL
- Supabase
- 其他可持久化后端

### 数据来源

完整模拟炒股研究至少需要：

- 股票基础行情
- RSI 数据
- 财务数据
- 资金面数据
- 催化剂 / 新闻 / 事件
- 题材信息
- 风险点材料

当前仓库可以先桥接已有 markdown 输出：

- 候选池：`candidate_list.md`
- 七层研究报告：Hermes 当前输出

## 最小可运行层级

### Level 0：本地安装预检

需要：

- Python 3.11+
- Git
- 本仓库

可运行：

- `scripts/bootstrap.ps1`
- `scripts/bootstrap.sh`
- `stock-agent-orchestrator doctor`
- `stock-agent-orchestrator demo`

能证明：

- 环境能装
- 包能导入
- 本地任务引擎能跑

不能证明：

- 飞书协同可用
- 用户能核实流程
- 多 agent 能透明闭环

### Level 1：飞书只读 Shadow

需要：

- 飞书群消息读取能力
- Orchestrator 后台服务
- 任务数据库

可运行：

- 自动识别委托
- 自动建任务
- 自动推断状态
- 生成断点报告

不能做：

- 主动发言
- 接管正式任务
- 修改正式 agent 配置

### Level 2：飞书 Beta 主动协同

需要：

- beta 飞书群
- 小C-beta
- 小智-beta
- 小巴-beta
- Orchestrator 后台服务
- 任务数据库
- 基础数据源或可替代样本

可运行：

- 小C-beta 建任务卡
- 小智-beta 回填七层数据
- 小巴-beta 输出分析结论
- 小C-beta 追办、请求审批、收口
- 用户在群里核实完整流程

这是产品可用性的第一道真实验收。

### Level 3：正式群灰度

需要：

- Level 2 稳定通过
- 明确审批边界
- 明确发送频率
- 明确异常回滚方式

可运行：

- 正式小C 接入 task-owner 能力
- 正式小智 / 小巴 保持原职责
- 已知规则内自动推进
- 新规则等待用户审批

## 最小验收清单

进入 beta 主动测试前，至少要准备：

- 一个飞书 beta 群
- 三个可区分的 agent 身份：小C-beta / 小智-beta / 小巴-beta
- 一个用户审批人
- 一个任务数据库
- 一套候选池输入方式
- 一套七层数据输入方式
- 一套任务卡展示格式
- 一套 `WAITING_USER` 规则
- 一套不刷屏的发送频率限制

## 不满足前提时的处理

如果暂时没有真实小智：

- 可以用七层样本文件代替
- 只能算数据层模拟

如果暂时没有真实小巴：

- 可以用分析样本文件代替
- 只能算分析层模拟

如果暂时没有飞书发送权限：

- 只能做只读 Shadow
- 不能进入 beta 主动测试

如果暂时没有数据源：

- 只能跑流程演示
- 不能验证选股质量

## 后续要补的配置文件

建议后续新增：

- `config/agents.example.yaml`：三 agent 身份与职责
- `config/feishu.example.yaml`：飞书群、机器人、权限配置
- `config/policy.example.yaml`：审批边界和自动推进规则
- `config/data_sources.example.yaml`：行情、RSI、财务等数据源配置

当前仓库已先提供 TOML 示例：

- `configs/beta.example.toml`
- `configs/formal.example.toml`

可用以下命令做接入前校验：

```bash
stock-agent-orchestrator validate-config --config configs/beta.example.toml
```

也可以先跑 fake 飞书连接器 smoke test：

```bash
stock-agent-orchestrator beta-smoke --config configs/beta.example.toml
```

这只会使用 `FakeFeishuClient` 记录发送请求，不会触达真实飞书群。
