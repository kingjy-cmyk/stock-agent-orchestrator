# 中文迭代路线图

## 目的

这个文档不是产品宣传页，而是后续真正推进仓库时的执行路线图。

你后面只需要做两件事：

1. 审核我提出的当前阶段方案
2. 决定是否进入下一阶段

## 当前总目标

把 `stock-agent-orchestrator` 做成一个成熟、完整、可持续维护的多 agent 模拟炒股研究系统。

当前明确只做：

- 研究
- 模拟盘
- 闭环任务推进
- 规则复盘与经验沉淀

当前不做：

- 实盘自动下单
- 未经审批的新规则自动生效
- 直接打扰正式群的高风险主动动作

## 阶段总览

### Phase 0：仓库骨架

当前状态：`已完成`

目标：

- 建立任务状态机
- 建立 CLI
- 建立候选池 / 七层报告桥接
- 建立规则建议生成
- 建立基础测试和文档

验收标准：

- 本地可安装
- 核心命令可跑
- 单元测试通过
- README 和中文文档齐全
- 运行前提与最小配置说明清楚

### Phase 1：Shadow Mode

当前状态：`Phase 1A 已开始`

目标：

- 先用离线消息样本做安装和解析预检
- 核心目标是接入飞书消息流做只读 Shadow
- 自动识别委托并建任务
- 不发消息，只做记录和状态推断
- 找出“为什么会没下文”的真实断点

要实现的能力：

- 历史消息样本导入
- codex-remote relay 日志脱敏抽样
- Feishu 消息转任务事件
- 群消息时间线落盘
- 超时未闭环检测
- `WAITING_USER` 与“静默断链”区分

验收标准：

- 能导入一段真实群消息样本
- 能生成对应任务和状态流转
- 能指出任务卡在哪一步
- 不影响正式群现有行为

需要你审批：

- 是否允许从真实日志中抽取样本
- 样本是否可以脱敏后保存在仓库或只放本地 `.runtime`
- 是否建立独立飞书 beta 群用于主动透明流程验收

### Phase 1A-3：任务聚合与拆装即用

当前状态：`已完成`

目标：

- 减少同一条委托被拆成多个碎片任务
- 提供新电脑安装后的自检命令
- 提供安装预检用的内置演示样本

已实现：

- `doctor` 环境自检
- `demo` 内置样本一键回放
- `scripts/bootstrap.ps1` 新电脑 Windows 一键启动
- `scripts/bootstrap.sh` macOS / Linux 一键启动
- “继续/下一步/修正/完善/推进”等跟进消息会并入当前任务
- 120 条真实 relay 日志离线回放中，任务碎片从约 30 个降到 16 个

验收标准：

- 新电脑 clone 后可以直接运行 bootstrap 脚本
- `stock-agent-orchestrator doctor` 通过
- `stock-agent-orchestrator demo` 能生成报告
- 真实日志回放的任务碎片数量持续下降
- 明确 demo 不是产品验收，产品验收必须回到飞书

### Phase 2：七层研究卡标准化

当前状态：`待开始`

目标：

- 把 Hermes 当前七层输出统一成仓库内的标准研究卡结构
- 固定现行七层 schema，不再漂移

固定 schema：

1. 行情
2. RSI
3. 财务
4. 资金面
5. 催化剂
6. 题材
7. 风险点

验收标准：

- 能从现有七层报告解析为统一结构
- 能输出一张标准研究卡
- 缺失字段会明确标红或标记空缺

需要你审批：

- 最终研究卡展示格式
- 哪些字段必须出现

### Phase 3：日常闭环联通

当前状态：`待开始`

目标：

- 把每日候选池和单票研究任务串成完整闭环
- 让 `小C` 真正具备持续追办能力

要实现的能力：

- 候选池导入后自动生成研究子任务
- 七层数据到达后自动进入分析阶段
- 分析完成后自动检查是否需要规则沉淀

验收标准：

- 一条日常链路能从候选池跑到研究结论
- 任务不会中途静默丢失
- 卡住时会明确进入 `WAITING_USER`

需要你审批：

- 哪个节点算真正 `CLOSED`
- 哪些分析结论允许自动收口

### Phase 4：Rule Memory 正式化

当前状态：`待开始`

目标：

- 把经验总结从“手动写教训”升级成“系统生成建议 + 你审批后固化”

要实现的能力：

- 自动抽取重复信号
- 生成规则建议草案
- 区分“强化旧规则”和“新规则提案”

验收标准：

- 完成任务后能自动生成规则建议
- 新规则不会绕过你直接生效
- 规则建议有证据来源

需要你审批：

- 哪类建议可以自动落盘
- 哪类建议必须人工审阅

### Phase 5：Beta 主动测试

当前状态：`待开始`

目标：

- 建 `小C-beta / 小智-beta / 小巴-beta`
- 在测试群里验证主动追办、自动推进和回执逻辑

验收标准：

- beta 组能完整跑一条日常研究链
- 不会刷屏
- 不会重复催办
- 不会把状态回执误判成新任务
- 用户能在飞书群里看到任务卡、状态、责任人、证据和审批点

需要你审批：

- beta 三角色命名
- beta 群消息边界
- 主动催办频率

### Phase 6：正式灰度

当前状态：`待开始`

目标：

- 先把新任务 owner 逻辑灰度接入正式 `小C`
- 后续再逐步挂接正式 `小智 / 小巴`

验收标准：

- 正式群体验不退化
- 小C 不再“做着做着没下文”
- 出现阻塞时能明确告诉你为什么卡住

需要你审批：

- 正式群允许哪些自动催办动作
- 何时开放更主动的任务接管

## 每轮推进方式

从现在开始，后续每一轮都按这个格式推进：

1. 我先给你本轮目标、改动点、风险和审批点
2. 你决定是否进入本轮
3. 我落代码、补文档、跑验证
4. 我把结果推到仓库
5. 你验收后决定下一轮

## 当前建议

当前最合理的下一轮就是：

`进入 Phase 1：Shadow Mode`

原因很简单：

- 风险最低
- 最贴近现有正式体系
- 能最快暴露“任务为什么没有闭环”的真实问题
- 不会直接破坏现在的小C / 小智 / 小巴 协同
