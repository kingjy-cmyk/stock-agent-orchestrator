# 飞书优先原则

## 结论

这个项目的主体验证平台必须是 `飞书群`。

`doctor`、`demo`、离线 Shadow 回放只用于安装预检和低风险调试，不能替代飞书里的真实透明协同验证。

## 为什么必须走飞书

用户真正需要核实的不是“程序能不能跑”，而是：

- 小C 是否持续拥有任务
- 小智 / 小巴 的协同过程是否透明
- 每一步状态是否能在群里被看见
- 卡住时是否明确告诉用户缺什么
- 任务是否有证据落盘，而不是只发一句话

这些都必须在飞书群这个控制面里验证。

## 正确分层

### 安装预检层

用途：新电脑能不能装、能不能启动、核心包能不能跑。

工具：

- `scripts/bootstrap.ps1`
- `scripts/bootstrap.sh`
- `stock-agent-orchestrator doctor`
- `stock-agent-orchestrator demo`

验收人：开发者。

结论边界：只能证明环境可用，不能证明产品可用。

### 飞书 Shadow 层

用途：只读接入飞书消息流，自动建任务、推断状态、生成报告。

行为：

- 读取群消息
- 建任务
- 推断状态
- 生成本地或后台报告
- 不主动发言
- 不改变正式小C / 小智 / 小巴 的行为

验收人：用户和开发者。

结论边界：能验证识别和断点分析，但还不能验证主动协同。

### 飞书 Beta 层

用途：在测试群里让 `小C-beta / 小智-beta / 小巴-beta` 主动发言，验证完整透明流程。

行为：

- 小C-beta 在群里建任务卡
- 小智-beta 回填七层数据状态
- 小巴-beta 回填分析状态
- 小C-beta 持续追办、提醒、收口
- 用户在群里审批新规则和越界动作

验收人：用户。

结论边界：这是产品可用性的第一道真实验收。

### 正式群灰度层

用途：把稳定后的 `小C` task-owner 能力逐步接入正式群。

行为：

- 优先升级小C 的追办和状态透明能力
- 不先改小智 / 小巴 的核心职责
- 不绕过用户审批新增规则
- 不直接进入实盘动作

验收人：用户。

## 飞书里必须可见的内容

每个任务至少要能在群里看见：

- 任务编号
- 当前目标
- 当前状态
- 当前责任人
- 下一步动作
- 已有证据
- 是否等待用户审批

## 产品验收标准

一个流程只有同时满足这些条件，才算通过：

- 用户能在飞书群里看到任务从委托到收口
- 小C 不会做着做着没下文
- 小智 / 小巴 的输出能挂到任务证据里
- 新规则不会绕过用户审批
- 卡住时群里能明确显示卡点
- 最终结果能落盘，并能回查
